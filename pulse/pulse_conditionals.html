条件分岐

より面白いプログラムを書くには、いくつかの制御構文が必要です。この章では、if/else と match の 2 種類の分岐を使ったプログラムを書いていきます。

簡単な分岐プログラム：最大値

2 つの参照に格納された値の最大値を返す簡単なプログラムを示します。

let max_spec x y = if x < y then y else x

fn max #p #q (x y:ref int)
requires pts_to x #p 'vx ** pts_to y #q 'vy
returns n:int
ensures pts_to x #p 'vx ** pts_to y #q 'vy
        ** pure (n == max_spec 'vx 'vy)
{
    let vx = !x;
    let vy = !y;
    if (vx > vy)
    {
        vx
    }
    else
    {
        vy
    }
}

このプログラムはごく一般的な仕様記述スタイルを示しています。

  • まず、純粋な F* 関数 max_spec を用意します。

  • 次に、可変参照を扱う Pulse の関数を定義し、純粋な F* の仕様と対応付ける仕様を書きます。ここでは、2 つの参照内容を表す論理的証人に対して、max が max_spec と同じ振る舞いをすることを証明します。

max の実装は Pulse の条件文を使います。これは F* の if-then-else 式とは文法が異なり、C のような言語に近い、中括弧を用いたより命令的な構文になっています。

制限事項：末尾でない条件分岐

現状の Pulse の推論機構は、末尾（tail）以外の位置に現れる条件分岐をまだサポートしていません。例えば次の max の変種は、以下のエラーメッセージと共に失敗します。

fn max_alt #p #q (x y:ref int)
requires pts_to x #p 'vx ** pts_to y #q 'vy
returns n:int
ensures pts_to x #p 'vx ** pts_to y #q 'vy
        ** pure (n == max_spec 'vx 'vy)
{
    let mut result = 0;
    let vx = !x;
    let vy = !y;
    if (vx > vy)
    {
        result := vx;
    }
    else
    {
        result := vy;
    };
    !result;
}

Pulse cannot yet infer a postcondition for a non-tail conditional statement;
Either annotate this `if` with `returns` clause; or rewrite your code to use a tail conditional

こちらは注釈を付けて通るようにした max_alt の例です。

fn max_alt2 #p #q (x y:ref int)
requires pts_to x #p 'vx ** pts_to y #q 'vy
returns n:int
ensures pts_to x #p 'vx ** pts_to y #q 'vy
        ** pure (n == max_spec 'vx 'vy)
{
    let mut result = 0;
    let vx = !x;
    let vy = !y;
    if (vx > vy)
    ensures
     exists* r.
       pts_to x #p 'vx **
       pts_to y #q 'vy **
       pts_to result r **
       pure (r == max_spec 'vx 'vy)
    {
        result := vx;
    }
    else
    {
        result := vy;
    };
    !result;
}

末尾でない条件分岐への推論サポートは現在開発中です。

NULL 可能参照でのパターンマッチ

パターンマッチの使い方を示すため、NULL の可能性がある参照の次の表現を考えます。

let nullable_ref (a: Type0) = option (ref a)

let pts_to_or_null #a
        (x:nullable_ref a) 
        (#[default_arg (`1.0R)] p:perm) //implicit argument with a default
        (v:option a)
: slprop
= match x with
  | None -> pure (v == None)
  | Some x -> exists* w. pts_to x #p w ** pure (v == Some w)

表現述語

NULL 可能参照は、表現述語 pts_to_or_null を伴う option (ref a) として表せます。いくつか注意点があります。

  • (#[default_arg (\`1.0R)] p:perm) という記法は、省略時に 1.0R を既定値とする暗黙引数を表す F* の構文です。Pulse.Lib.Reference.pts_to などの述語も同様に定義されています。

  • 定義は場合分けになっています。参照 x が None なら、論理的証人も None です。

  • それ以外の場合、基になる参照は何らかの値 w を指しており、論理的証人は v == Some w となって値と一致します。

次のように定義したくなるかもしれません。

let pts_to_or_null #a
     (x:nullable_ref a)
     (#[default_arg (`1.0R)] p:perm)
     (v:option a)
: slprop
= match x with
  | None -> pure (v == None)
  | Some x -> pure (Some? v) ** pts_to x #p (Some?.v v)

しかし、F* の連言 p /\ q``（ここでは ``q の型付けに p を利用できる）と異なり、** は左に偏った演算子ではありません。したがってこの文脈では (Some?.v v) を示せず、この定義は却下されます。

別の書き方として次のようなスタイルも考えられます。

let pts_to_or_null #a
     (x:nullable_ref a)
     (#[default_arg (`1.0R)] p:perm)
     (v:option a)
: slprop
= match x, v with
  | None, None -> emp
  | Some x, Some w -> pts_to x #p w
  | _ -> pure False

これでも動きますが、（実際には起こりえない）追加のケース処理が必要になります。

NULL 可能参照の読み出し

Pulse で最初のパターンマッチを書いてみましょう。

fn read_nullable #a #p (r:nullable_ref a)
requires pts_to_or_null r #p 'v
returns o:option a
ensures pts_to_or_null r #p 'v
        ** pure ('v == o)
{
    match r {
     Some x -> {
        unfold (pts_to_or_null (Some x) #p 'v);
        let o = !x;
        fold (pts_to_or_null (Some x) #p 'v);
        rewrite each (Some x) as r;
        Some o
     }
     None -> {
        unfold (pts_to_or_null None #p 'v);
        fold (pts_to_or_null None #p 'v);
        rewrite each (None #(ref a)) as r;
        None
     }
    }
}

Pulse のパターンマッチの構文は、F* よりも命令的で、Rust に近い書き味です。

  • match の本体全体を中括弧で囲みます。

  • 各分岐もそれぞれ中括弧で囲みます。

  • （現時点の）Pulse がサポートするのは、トップレベルにコンストラクタ 1 つ＋変数、または変数のみという単純なパターンです。例えば Some (Some x) のようなパターンは書けません。

read_nullable の型は、表現述語の論理的証人と等しい値を返すことを約束します。

コードは少し冗長です——すぐ後で、もう少しすっきり書く方法を見ます。

Some x 側の分岐で show_proof_state を呼ぶと、次のように表示されます。

- Current context:
    pts_to_or_null r (reveal 'v)
- In typing environment:
    [branch equality#684 : squash (eq2 r (Some x)),
     ...

注目すべきは branch equality という仮定で、この分岐では (r == Some x) を仮定できるという意味です。そこで、まず r を書き換え、表現述語を unfold し、x から値 o を読み、述語を fold で戻し、逆方向へ書き換え、最後に Some o を返します。None 側も同様です。

Pulse の match は、F* と異なり、否定された経路条件を与えません。例えば次の例では、パターンがワイルドカードのみで、未選択の前半の分岐に対する経路条件として not (Some? x) をPulse チェッカが仮定しないため、主張が失敗します。

[@@expect_failure]
fn read_nullable_alt #a #p (r:nullable_ref a)
requires pts_to_or_null r #p 'v
returns o:option a
ensures emp
{
    match r {
     Some x -> { admit () }
     _ -> { 
        // we only have `r == _` in scope
        // not the negation of the prior branch conditions
        // i.e., unlike F*, we don't have not (Some? r)
        // so the assertion below fails
        assert (pure (r == None)); 
        admit() }
    }
}

これらの否定的な経路条件も提供できるよう、Pulse チェッカの拡張を予定しています。

ヘルパー

pts_to_or_null のように slprop が場合分けで定義されていると、パターンマッチ時にもその場合分けに沿って推論することがよくあります。毎回書き換え→unfold→fold→書き換え…を行う代わりに、これらを処理するヘルパー関数を定義できます。

ghost
fn elim_pts_to_or_null_none #a #p (r:nullable_ref a)
requires pts_to_or_null r #p 'v ** pure (r == None)
ensures pts_to_or_null r #p 'v ** pure ('v == None)
{
    rewrite each r as None;
    unfold (pts_to_or_null None #p 'v);
    fold (pts_to_or_null None #p 'v);
    rewrite each (None #(ref a)) as r;
}

ghost
fn intro_pts_to_or_null_none #a #p (r:nullable_ref a)
requires pure (r == None)
ensures pts_to_or_null r #p None
{
    fold (pts_to_or_null #a None #p None);
    rewrite each (None #(ref a)) as r;
}

ghost
fn elim_pts_to_or_null_some #a #p (r:nullable_ref a) (x:ref a)
requires pts_to_or_null r #p 'v ** pure (r == Some x)
ensures exists* w. pts_to x #p w ** pure ('v == Some w)
{
    rewrite each r as (Some x);
    unfold (pts_to_or_null (Some x) #p 'v);
}

ghost
fn intro_pts_to_or_null_some #a #p (r:nullable_ref a) (x:ref a)
requires pts_to x #p 'v ** pure (r == Some x)
ensures pts_to_or_null r #p (Some 'v)
{
    fold (pts_to_or_null (Some x) #p (Some 'v));
    rewrite each (Some x) as r;
}

これらの関数はすべて ghost としてマークされ、純粋に証明目的でのみ用いられることを示します。

この種のヘルパーは多くの場合、機械的に作れます。将来的には pts_to_or_null の定義からPulse チェッカが自動生成することも考えられますし、F* のメタプログラミングで独自に自動生成することも可能です。現時点では手書きします。

ヘルパーを使うと、NULL 可能参照のケース分解がいくらか楽になります：

fn read_nullable_alt #a #p (r:nullable_ref a)
requires pts_to_or_null r #p 'v
returns o:option a
ensures pts_to_or_null r #p 'v
        ** pure ('v == o)
{
    match r {
     Some x -> {
        elim_pts_to_or_null_some (Some x) x;
        let o = !x;
        intro_pts_to_or_null_some r x;
        Some o
     }
     None -> {
        unfold pts_to_or_null None 'v;
        fold pts_to_or_null None 'v;
        None
     }
    }
}

NULL 可能参照への書き込み

ヘルパーを定義したら、繰り返し利用できます。例えば、NULL 可能参照へ書き込む関数を次に示します。

fn write_nullable #a (r:nullable_ref a) (v:a)
requires pts_to_or_null r 'v
ensures exists* w. pts_to_or_null r w ** pure (Some? r ==> w == Some v)
{
    match r {
     None -> {
        rewrite pts_to_or_null None 'v
             as pts_to_or_null r 'v;
     }
     Some x -> {
        unfold pts_to_or_null (Some x) 'v;
        x := v;
        intro_pts_to_or_null_some r x;
     }
    }
}
