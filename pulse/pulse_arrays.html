可変配列

この章では Pulse における可変配列を学びます。配列は同一型の値が連続して並ぶ集合です。ref と同様に、Pulse の配列は現在の関数のスタックフレーム上、またはヒープ上に確保できます——スタック確保された配列は（関数の復帰時などに）自動的に回収されますが、ヒープ確保された配列はプログラマが明示的に管理します。

Pulse には 2 種類の配列型があります。基本的な配列型である Pulse.Lib.Array.array t と、ヒープ確保用の Pulse.Lib.Vec.vec t です。コード再利用のため、スタック配列／ヒープ配列の両方に対して動作する関数は Pulse.Lib.Array.array t を用いて記述できます——Pulse.Lib.Vec ライブラリが vec t と array t の相互変換（コアーション）を提供します。

array t

配列を読み出す次の例を使って、array t の基本を説明します。

fn read_i
  (#[@@@ Rust_generics_bounds ["Copy"]] t:Type0)
  (arr:array t)
  (#p:perm)
  (#s:erased (Seq.seq t))
  (i:SZ.t { SZ.v i < Seq.length s })
  requires pts_to arr #p s
  returns x:t
  ensures pts_to arr #p s ** pure (x == Seq.index s (SZ.v i))
{
  arr.(i)
}

ライブラリは pts_to arr #p s という points-to 述語を提供します。これは現在のメモリにおいて arr の内容が（関数型）列 s:FStar.Seq.seq t と同一であることを表します。参照に対する pts_to と同様、暗黙の分数パーミッション p を添字に取り、共有の読み取り専用アクセスと、排他的読み書きアクセスを区別します。

read_i の引数中の `s は仕様専用のため消去（erasure）されます。

配列の読み書きは C の size_t を F* で表した FStar.SizeT.t 型のインデックスで行います [1]。インデックスが配列境界内であることは、module SZ = FStar.SizeT として SZ.v i < Seq.length s の絞り込みで保証します。関数は配列の i 番目の要素を返し、事後条件の slprop pure (x == Seq.index s (SZ.v i)) でそれを述べます。本体では配列読取演算子 arr.(i) を用います。

別の例として、配列の i 番目の要素への書き込みを見てみましょう。

fn write_i (#t:Type0) (arr:array t) (#s:erased (Seq.seq t)) (x:t) (i:SZ.t { SZ.v i < Seq.length s })
  requires pts_to arr s
  ensures pts_to arr (Seq.upd s (SZ.v i) x)
{
  arr.(i) <- x
}

この関数は配列書込演算子 arr(i) <- x を使い、復帰時の状態で配列内容が、インデックス i を更新した列 s と同一であることを事後条件で主張します。

読み取りは任意のパーミッションで可能ですが、書き込みには 1.0R``（排他的権限）が必要です。例えば、``1.0R なしの write_i 実装は次のように却下されます。

[@@ expect_failure]

fn write_ip #t (arr:array t) (#p:perm) (#s:erased _) (x:t) (i:SZ.t { SZ.v i < Seq.length s })
  requires pts_to arr #p s
  ensures pts_to arr #p (Seq.upd s (SZ.v i) x)
{
  arr.(i) <- x
}

ライブラリには参照の場合と同様に、配列のパーミッションを分割・合成する share と gather が用意されています。

ここからは、条件分岐・ループ・存在量化・不変条件など、これまでに見た Pulse の構成要素を用いて配列を扱う例をいくつか見ていきます。

[1]

C の size_t は少なくとも 16 ビット幅の符号なし整数型で、その上限はプラットフォーム依存です。FStar.SizeT.size_t はこの型をモデル化し、前述の 有界整数型 同様、C ではプリミティブな size_t に抽出されます。

比較

2 つの配列が等しいかどうかを比較する関数を実装してみましょう。

fn compare
  (#[@@@ Rust_generics_bounds ["PartialEq"; "Copy"]] t:eqtype)
  #p1 #p2
  (a1 a2:A.array t)
  (l:SZ.t) 
  requires (
    A.pts_to a1 #p1 's1 **
    A.pts_to a2 #p2 's2 **
    pure (Seq.length 's1 == Seq.length 's2 /\ Seq.length 's2 == SZ.v l)
  )
  returns res:bool
  ensures (
    A.pts_to a1 #p1 's1 **
    A.pts_to a2 #p2 's2 **
    pure (res <==> Seq.equal 's1 's2)
  )

この関数は 2 つの配列 a1 と a2 を受け取り、真偽値を返します。事後条件 pure (res <==> Seq.equal 's1 's2) は、両配列の列表現が等しい場合に限り結果が真になることを示します。配列は読み取りのみなので、各配列のパーミッション p1 と p2 に関して関数は多相です。型引数 t は eqtype で、t の値が決定可能な等値判定を持つことを要求します。

compare の実装方法の一つは、可変カウンタで 2 つの配列を走査する while ループを用い、対応する要素が等しいか逐次チェックすることです。

{
  let mut i = 0sz;
  while (
    let vi = !i;
    if (vi <^ l) {
      (a1.(vi) = a2.(vi))
    } else {
      false
    }
  )
  invariant
  exists* (vi:_{SZ.v vi <= SZ.v l}). ( 
    R.pts_to i vi **
    A.pts_to a1 #p1 's1 **
    A.pts_to a2 #p2 's2 **
    pure (
      (forall (i:nat). i < SZ.v vi ==> Seq.index 's1 i == Seq.index 's2 i)
    )
  )
  {
    i := !i +^ 1sz
  };
  (!i = l);
}

ループ不変条件は次を述べます。(a) カウンタの現在値未満の範囲で配列が要素ごとに等しいこと、(b) 真偽値 b は、現在のカウンタ値が配列長未満で、かつその位置の要素が等しい場合に限り真であること。(a) は compare の最終的な事後条件の証明に役立ち、(b) はループ本体でカウンタを増やした後も不変条件を保つために必要です。

コピー

次の例として、配列 a2 の内容を a1 にコピーする copy 関数を実装しましょう。

fn copy
  (#[@@@ Rust_generics_bounds ["Copy"]] t:Type0)
  (a1 a2:A.array t)
  (l:SZ.t)
  (#p2:perm)
  requires (
    A.pts_to a1 's1 **
    A.pts_to a2 #p2 's2 **
    pure (Seq.length 's1 == Seq.length 's2 /\ Seq.length 's2 == SZ.v l)
  )
  ensures (
    (exists* s1. A.pts_to a1 s1 ** pure (Seq.equal s1 's2)) **
    A.pts_to a2 #p2 's2
  )
{
  let mut i = 0sz;
  while (
    (!i <^ l)
  )
  invariant
  exists* vi s1. ( 
    R.pts_to i vi **
    A.pts_to a1 s1 **
    A.pts_to a2 #p2 's2 **
    pure (
      Seq.length s1 == Seq.length 's2 /\
      SZ.v vi <= SZ.v l /\
      (forall (i:nat). i < SZ.v vi ==> Seq.index s1 i == Seq.index 's2 i)
    )
  )
  {
    let vi = !i;
    a1.(vi) <- a2.(vi);
    i := vi +^ 1sz
  }
}

ループ不変条件は a1 の内容を存在的に抽象化し、現在のカウンタ位置までの内容が 2 配列で等しいことを保ちます。残りのコードは素直で、ループ条件でカウンタが配列長未満であることを確認し、ループ本体で 1 要素ずつコピーします。

copy の事後条件はやや回りくどいことに気づくでしょう。より良いシグネチャとして、a1 の内容が直接 's2 と同一だと述べる次の形が考えられます。

fn copy2
  (#[@@@ Rust_generics_bounds ["Copy"]] t:Type0)
  (a1 a2:A.array t)
  (l:SZ.t)
  (#p2:perm)
  requires (
    A.pts_to a1 's1 **
    A.pts_to a2 #p2 's2 **
    pure (Seq.length 's1 == Seq.length 's2 /\ Seq.length 's2 == SZ.v l)
  )
  ensures (
    A.pts_to a1 's2 **
    A.pts_to a2 #p2 's2
  )

このシグネチャでも実装可能ですが、while ループの後で事後条件をこの形に整えるための書き換えを一手入れる必要があります。

  // after the loop
  with v1 s1. _; //bind existentially bound witnesses from the invariant
  Seq.lemma_eq_elim s1 's2; //call an F* lemma to prove that s1 == 's2
  ()

前の章 で見たように、述語を書き換える明示的な方法を使うこともできます。

スタック確保配列

スタック配列は [| v; n |] という式で確保できます。サイズ n の配列を確保し、全要素を v で初期化します。n はコンパイル時定数でなければなりません。生成された配列が長さ n の v の列を指すことが事後条件として与えられます。次の例では、2 つの配列をスタックに確保し、上の compare 関数で比較します。

fn compare_stack_arrays ()
  requires emp
  ensures emp
{
  // |- emp
  let mut a1 = [| 0; 2sz |];
  // a1:array int |- pts_to a1 (Seq.create 0 (SZ.v 2))
  let mut a2 = [| 0; 2sz |];
  // a1 a2:array int |- pts_to a1 (Seq.create 0 (SZ.v 2)) ** pts_to a2 (Seq.create 0 (SZ.v 2))
  let b = compare a1 a2 2sz;
  assert (pure b)
}

スタック参照と同様、スタック配列は明示的な解放や drop は不要で、関数の復帰時に自動で回収されます。そのため、関数の戻り値として返すことはできません。

[@@ expect_failure]

fn ret_stack_array ()
  requires emp
  returns a:array int
  ensures pts_to a (Seq.create 2 0)
{
  let mut a1 = [| 0; 2sz |];
  a1  // cannot prove pts_to a (Seq.create 0 2) in the context emp
}

ヒープ確保配列

Pulse.Lib.Vec ライブラリはヒープ確保配列用の型 vec t を提供します。関係性で言えば、vec と array は、box と ref の関係に対応します。

array と同様、vec にも分数パーミッション対応の pts_to 述語、パーミッション分割／合成の share／gather、および読み書き関数が備わります。ただし array と異なり、Vec ライブラリは確保と解放の関数も提供します。

module V = Pulse.Lib.Vec

 
fn heap_arrays ()
  requires emp
  returns a:V.vec int
  ensures V.pts_to a (Seq.create 2 0)
{
  let a1 = V.alloc 0 2sz;
  let a2 = V.alloc 0 2sz;
  V.free a1;
  a2  // returning vec is ok
}

ヒープ参照の場合と同様に、ヒープ配列はコアーション vec_to_array により array へ変換できます。この変換を使う際には、多くの場合 Vec.pts_to と Array.pts_to の相互変換が必要になります。そのための補題として to_array_pts_to と to_vec_pts_to が用意されています。

次の例はこのパターンを示します。先ほど作成した copy2 関数を用いて、スタック配列の内容をヒープ配列へコピーします。

fn copy_app ([@@@ Rust_mut_binder] v:V.vec int)
  requires exists* s. V.pts_to v s ** pure (Seq.length s == 2)
  ensures V.pts_to v (Seq.create 2 0)
{
  let mut a = [| 0; 2sz |];
  // v, s |- V.pts_to v s ** ...
  V.to_array_pts_to v;
  // v, s |- pts_to (vec_to_array v) s ** ...
  copy2 (V.vec_to_array v) a 2sz;
  // v, s |- pts_to (vec_to_array v) (Seq.create 2 0) ** ...
  V.to_vec_pts_to v
  // v, s |- V.pts_to v (Seq.create 2 0) ** ...
}

v に関するアサーションが V.pts_to から配列用の pts_to へ、そしてまた戻る点に注意してください。これは、配列アルゴリズムやルーチンを array t で実装しておけば、スタック配列とヒープ配列の双方で再利用できることを意味します。

最後に、vec a という名前は Rust の std::Vec を想起させますが、現時点では自動リサイズは未対応です。
