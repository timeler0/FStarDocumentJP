ゴースト計算

純粋な F* の章を通じて、プログラムの性質を証明するために Lemma とゴースト関数を常用してきました。思い出していただくと、Lemma は常に unit を返す純粋かつ全域的な関数であり、計算的な意味を持たず F* コンパイラによって消去されます。F* の ゴースト関数 も純粋・全域的ですが、消去された値を制御下で参照できる点が異なります—これらも最終的にはコンパイラにより消去されます。

既に見たように、F* の lemma とゴースト関数は Pulse のコードから直接利用できます。しかしそれらは、スコープ内の純粋な値に関する性質を述べる場合に限って有用です。Pulse ではしばしば、状態について述べ、slprop を操作するような lemma が必要になります。そのために Pulse は独自の ゴースト計算*（F の lemma／ゴースト関数に相当しますが、slprop を用いて記述されるもの）と、ゴースト状態*（F の消去型に相当しますが、計算的には無意味でありつつ可変な状態）を提供します。ゴースト計算は Pulse の至るところで使われ—すでにいくつか例を見ました—特に並行プログラムの証明ではゴースト状態が有用です。

ゴースト関数

次の Pulse 関数は型検査に失敗し、以下のようなエラーになります。

[@@expect_failure]
fn incr_erased_non_ghost (x:erased int)
requires emp
returns y:int
ensures emp ** pure (y == x + 1)
{
  let x = reveal x;
  (x + 1)
}

Cannot bind ghost expression reveal x with ST computation

これは失敗して当然です。消去された整数 x をインクリメントして整数 y を計算できると主張しているからです。x:erased int は実行時には存在しないため、このプログラムはコンパイルできません。

しかし関数に ghost 修飾子を付ければ動作します。

ghost
fn incr_erased (x:erased int)
requires emp
returns y:int
ensures emp ** pure (y == x + 1)
{
  let x = reveal x;
  (x + 1)
}

ghost 修飾子は、その関数が実行時に消去されることを Pulse に示します。そのため FStar.Ghost.reveal のような GTot 効果の F* 関数を利用できます。

ただし健全性のため、コンパイルされるコードが ghost 関数の戻り値に依存してはなりません。したがって、次のコードはエラーになります。

[@@expect_failure]
fn use_incr_erased (x:erased int)
requires emp
returns y:int
ensures emp ** pure (y == x + 1)
{
  incr_erased x;
}

Expected a term with a non-informative (e.g., erased) type; got int

つまり、非ゴースト文脈から ghost 関数を呼ぶ場合、その戻り型は erased や unit といった非情報型である必要があります。非情報型のクラスと、F* の ゴースト計算を全域的文脈で使用可能にする規則 はここで説明されており、同じ規則が Pulse にも適用されます。

incr_erased を非ゴースト文脈で使うには、その結果を消去しなければなりません。方法はいくつかあります。

やや冗長ですが明示的な方法として、incr_erased の呼び出しを包む入れ子のゴースト関数を定義する手があります。

fn use_incr_erased (x:erased int)
requires emp
returns y:erased int
ensures emp ** pure (y == x + 1)
{
  ghost
  fn wrap (x:erased int)
  requires emp
  returns y:erased int
  ensures emp ** pure (y == x + 1)
  {
    let y = incr_erased x;
    hide y
  };
  wrap x
}

ライブラリにはゴースト呼び出しの結果を消去する高階コンビネータ Pulse.Lib.Pervasives.call_ghost も用意されています。

fn use_incr_erased_alt (x:erased int)
requires emp
returns y:erased int
ensures emp ** pure (y == x + 1)
{ 
  call_ghost incr_erased x;
}

call_ghost は引数個数の異なるゴースト関数にも使えますが、適用は次のようにカリー化されている必要があります。

例えば add_erased のような 2 引数のゴースト関数があるとします。

ghost
fn add_erased (x y:erased int)
requires emp
returns z:int
ensures emp ** pure (z == x + y)
{
  let x = reveal x;
  let y = reveal y;
  (x + y)
}

これを非ゴースト文脈で呼ぶには、次のようにします。

fn use_add_erased (x y:erased int)
requires emp
returns z:erased int
ensures emp ** pure (z == x + y)
{
  call_ghost (add_erased x) y
}

とはいえ、非ゴースト文脈で使えるようにするには結局戻り型を非情報型にする必要があるので、各呼び出し箇所でラップするより、最初からそのように定義しておくのが一般に良いでしょう。

ghost
fn add_erased_erased (x y:erased int)
requires emp
returns z:erased int
ensures emp ** pure (z == x + y)
{
  let x = reveal x;
  let y = reveal y;
  hide (x + y)
}

いくつかの基本的なゴースト関数

前後条件が emp や pure _ のようなゴースト関数はあまり面白くありません—それらは通常の F* のゴースト関数で書けるからです。

ゴースト関数はしばしば slprop 間の同値性を示す証明ステップとして用いられます。以前 ゴースト関数の例 をいくつか見ましたが、それらは Pulse ライブラリの ghost 関数の合成として実装されているためゴーストです。

以前 見た rewrite プリミティブも、実は Pulse ライブラリで定義された関数です。そのシグネチャは次のようになっています。

ghost
fn __rewrite (p q:slprop)
requires p ** pure (p == q)
ensures q

fold や unfold など他の多くのプリミティブも rewrite を用いて定義されており、ゴースト計算です。

introduce exists* のような他のプリミティブも、次のようなシグネチャを持つライブラリのゴースト関数として実装されています。

ghost
fn intro_exists (#a:Type0) (p:a -> slprop) (x:erased a)
requires p x
ensures exists* x. p x

再帰述語とゴースト補題

以前、データ構造の表現述語などを 独自の述語として定義 する方法を見ました。slprop は通常の型に過ぎないため、F* で再帰的に定義することもできます。Pulse でこれらの再帰述語を扱う際は、補題として再帰的なゴースト関数を書くのが一般的です。ここでは簡単な例を示し、連結リストのような無界構造を扱う後続章で再訪します。

参照のリストがあり、それぞれが高々 n の整数を保持していることを表したいとします。そのための再帰述語が all_at_most l n です。

let rec all_at_most (l:list (ref nat)) (n:nat)
: slprop
= match l with
  | [] -> emp
  | hd::tl -> exists* (i:nat). pts_to hd i ** pure (i <= n) ** all_at_most tl n

ヌル許容参照 のときと同様に、各場合分けに対応してこの述語を導入・除去する小さなゴースト関数群を用意しておくと便利です。

再帰的ゴースト補題

Pulse では、Pulse コードで使う補題として再帰的なゴースト関数を書くことができます。F* の補題と同様、再帰ゴースト関数は全ての入力で停止することを証明しなければなりません—さもないと健全ではありません。

実例として、n <= m なら all_at_most l n を all_at_most l m へ弱められることを示すゴースト関数を書きましょう。

ghost
fn rec weaken_at_most (l:list (ref nat)) (n:nat) (m:nat)
requires all_at_most l n ** pure (n <= m)
ensures all_at_most l m
decreases l
{
  match l {
    [] -> {
      unfold (all_at_most [] n);
      fold (all_at_most [] n);
    }
    hd :: tl -> {
      unfold (all_at_most (hd::tl) n);
      weaken_at_most tl n m;
      fold (all_at_most (hd::tl) m);
    }
  }
}

注意すべき点をいくつか：

  • Pulse での再帰関数は fn rec で定義します。

  • 再帰ゴースト関数には decreases 注釈も必要です—F* と異なり、Pulse はまだ既定の減少注釈を推論しません。この例ではリスト l による再帰です。

  • Pulse のリストパターンは（まだ）F* と同程度の糖衣構文を持ちません。つまり [] や hd::tl をそのままパターンとしては書けません。

  • 証明自体は比較的素直です。

    • Nil の場合、まず n での all_at_most を除去し、m で導入します。

    • Cons の場合、all_at_most l n を除去し、帰納法の仮定を用いて tl 上の述語を弱め、最後に hd の仮定と合わせて再導入します。

可変ゴースト参照

Pulse の基盤となるロジックは、可換部分モノイド（PCM）に基づく非常に一般的なゴースト状態をサポートします。ユーザは F* で PCM を使って独自のゴースト状態抽象を定義し、Pulse プログラムで利用できます。ライブラリ Pulse.Lib.GhostReference は最も単純で一般的な形式—消去値への参照と、分数パーミッションに基づく所有権規律—を提供します。

以下では module GR = Pulse.Lib.GhostReference というモジュール略称を使います。このライブラリは Pulse.Lib.Reference に非常によく似ており、次のものを提供します：

  • GR.ref a：ゴースト参照の主要な型。GR.ref は消去可能な型であり、非情報型として扱われます。

  • GR.pts_to (#a:Type0) (r:GR.ref a) (#p:perm) (v:a) : slprop：ライブラリが提供する主たる述語。通常の pts_to と同様、パーミッション引数の既定は 1.0R です。

  • ref a``（というより ``box a に近い）と異なり、ゴースト参照 GR.ref a は字句スコープに束縛されません。GR.alloc で確保し、GR.free で解放します。もちろん確保／解放は実行時コストを持たず、純粋にゴースト操作です。

  • r:GR.ref a のとき、!r による読み出しは erased a を返します。同様に、更新には新しい値 v:erased a を与えれば十分です。

  • ゴースト参照の share と gather も ref と同様に機能します。

やや作為的な例

ゴースト状態が必要となる例の多くは、複数スレッド間の興味深い不変量の記述や、逐次実行でも異なるコンポーネント間の知識の関連付けに関わります。これらは後の章で扱います。ここでは雰囲気を掴むための小さな例を示します。

関数に参照への読み書き権限を与えつつ、戻る前に参照の値を元に戻すことを保証したいとします。最も単純なのは、次のようなシグネチャを関数に与えることです。

fn uses_but_resets #a (x:ref a)
requires pts_to x 'v
ensures pts_to x 'v

これをゴースト参照を用いて実現する別の方法を示します。

まず、実体参照への全権限とゴースト参照への半分の権限を持ち、両者が同じ値を保持していることを強制する述語 correlated を定義します。

let correlated #a (x:ref a) (y:GR.ref a) (v:a)=
  pts_to x v ** GR.pts_to y #0.5R v

次に関数 use_temp のシグネチャを示します。これだけを見ると、事前条件で束縛された証人 v0 と事後条件で束縛される v1 は無関係に見えるかもしれません。

fn use_temp (x:ref int) (y:GR.ref int)
requires exists* v0. correlated x y v0
ensures exists* v1. correlated x y v1

しかし use_temp はゴースト参照に対して半分の権限しか持たず、そこは変更できません。したがって実体参照を変更できるとしても、事後条件を満たして戻るには、参照を初期値に戻さざるを得ません。

{
  unfold correlated u#0;
  let v = !x;
  x := 17; //temporarily mutate x, give to to another function to use with full perm
  x := v; //but, we're forced to set it back to its original value
  fold correlated u#0;
}

呼び出し側はこの性質を利用して、参照を use_temp に渡し、戻ってきたときに値が変わっていないことを保証できます。

fn use_correlated ()
requires emp
ensures emp
{
  let mut x = 17;
  let g = GR.alloc 17;
  GR.share g;
  fold correlated u#0;  // GR.pts_to g #0.5R 17 ** correlated x g 17
  use_temp x g;     // GR.pts_to g #0.5R 17 ** correlated x g ?v1
  unfold correlated u#0; // GR.pts_to g #0.5R 17 ** GR.pts_to g #0.5R ?v1 ** pts_to x ?v1
  GR.gather g;       //this is the crucial step
                     // GT.pts_to g 17 ** pure (?v1 == 17) ** pts_to x ?v1
  assert (pts_to x 17);
  GR.free g;
}
