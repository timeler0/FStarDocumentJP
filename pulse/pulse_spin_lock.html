スピンロック

アトミック操作と不変条件を用いれば、並行プログラミング向けに多くの有用な抽象化を構築できます。この章では、相互排他のためのスピンロックをどのように構築するかを見ていきます。

ロックの表現

実装の主眼は、可変な機械語ワードでロックを表すことです。値 0ul はロックが解放されていること、1ul は取得済みであることを表します。ロック取得では、1ul への設定に成功するまで CAS（Compare-And-Swap）を原子的に試行し続けます。解放はより簡単で、参照を 0ul に設定します（ただし二重解放への対処という微妙さについても後で触れます）。

仕様の観点から見ると、ロックは不変条件にとても似ています。述語型 lock_alive l p は、ロックが性質 p を保護していることを述べます。ロックの取得では呼び出し側に p を与え、解放では呼び出し側が p の所有権を手放すことを要求します。実行時の相互排他は、ロックが利用可能になるまで取得側がスピン（ループ）することで強制されます。

ロックは U32.t への参照と不変条件のペアとして表現します。

let maybe (b:bool) (p:slprop) =
  if b then p else emp

let lock_inv (r:B.box U32.t) (p:slprop) : slprop =
  exists* v. B.pts_to r v ** maybe (v = 0ul) p

noeq
type lock = {
  r:B.box U32.t;
  i:iname;
}

let lock_alive (l:lock) (p:slprop) =
  inv l.i (lock_inv l.r p)

述語 lock_inv r p は次を述べます。

  • r:box U32.t に対する全権限を保持していること。

  • r の内容が 0ul なら、同時に p も保持していること。

ロック自体は、具体的な可変状態 box U32.t と不変条件参照 i:iref の組です。述語 lock_alive l p は、l.i が lock_inv l.r p を表す不変条件の名前であることを述べます。

ロックの生成

ロック生成は以下の new_lock で行います。呼び出し側は p を提供し、その所有権を新たに確保された l:lock に譲渡します。

fn new_lock (p:slprop)
requires p
returns l:lock
ensures lock_alive l p
{
   let r = B.alloc 0ul;
   fold (maybe (0ul = 0ul) p);
   fold (lock_inv r p);
   let i = new_invariant (lock_inv r p);
   let l = {r;i};
   rewrite (inv i (lock_inv r p)) as
           (inv l.i (lock_inv l.r p));
   fold lock_alive;
   l
}

実装上の注意点：

  • Box.alloc で参照をヒープ確保します。ロックはこの関数のスコープ外でも存続する必要が明らかだからです。

  • new_invariant で inv i (lock_inv r p) を作り、新規確保した参照と組みにします。

ロックへの権限の複製

ロックは複数スレッドで共有できてこそ有用です。lock_alive l p はロックの所有を表しますが、lock_alive は単なる不変条件なので、dup_inv を用いて複製できます。

ghost
fn dup_lock_alive (l:lock) (p:slprop)
  requires lock_alive l p
  ensures lock_alive l p ** lock_alive l p
{
  unfold lock_alive;
  dup_inv l.i (lock_inv l.r p);
  fold lock_alive;
  fold lock_alive
}

ロックの取得

acquire のシグネチャは次のとおりです。lock_alive l p があれば、何も証明せずに p を取り出せる、すなわち事前条件は emp であることを述べます。

fn rec acquire (#p:slprop) (l:lock)
requires lock_alive l p
ensures lock_alive l p ** p

最初は意外に思えるかもしれません。しかし、p はロックに格納された不変条件の中に退避されていること、そして acquire はロック中の参照への CAS が成功するまでループし続け、成功したときに p を取り出して呼び出し側へ返すことを思い出してください。

CAS（Compare-And-Swap）の型は Pulse.Lib.Reference で次のように定義されています。

let cond b (p q:slprop) = if b then p else q

atomic
fn cas_box (r:Box.box U32.t) (u v:U32.t) (#i:erased U32.t)
requires Box.pts_to r i
returns b:bool
ensures cond b (Box.pts_to r v ** pure (reveal i == u))
               (Box.pts_to r i)

cas_box r u v の仕様は、r を u から v へ原子的に更新しようと試み、成功した場合、r の初期値（i）が u に等しかったことが分かる、と述べています。

cas_box を用いて、acquire は末尾再帰関数として実装できます。

{
  unfold lock_alive;
  later_credit_buy 1;
  let b = 
    with_invariants l.i
      returns b:bool
      ensures later (lock_inv l.r p) ** maybe b p
      opens [l.i] {
      later_elim _;
      unfold lock_inv;
      let b = cas_box l.r 0ul 1ul;
      if b
      { 
        elim_cond_true _ _ _;
        with _b. rewrite (maybe _b p) as p;
        fold (maybe false p);
        rewrite (maybe false p) as (maybe (1ul = 0ul) p);
        fold (lock_inv l.r p);
        fold (maybe true p);
        later_intro (lock_inv l.r p);
        true
      }
      else
      {
        elim_cond_false _ _ _;
        fold (lock_inv l.r p);
        fold (maybe false p);
        later_intro (lock_inv l.r p);
        false
      }
    };
  fold lock_alive;
  if b { rewrite (maybe b p) as p; }
  else { rewrite (maybe b p) as emp; acquire l }
}

実装の要は with_invariants ブロックです。

  • その戻り値型は b:bool、事後条件は inv l.i (lock_inv l.r p) ** maybe b p です。これは 1 回の cas の後、成功したなら p を得られ、同時に不変条件を維持していることを表します。

  • l.i を開いて lock_inv を得てから、cas_box l.r 0ul 1ul を試みます。

  • cas_box が成功した場合、ロックは初期状態で 0ul だったと分かります。よって lock_inv から p を得て、それをロックから“取り出し”、maybe true p としてブロック外へ返せます。重要なのは、現在の値が 1ul であることが分かっているため、不変条件 lock_inv は自明に復元できる点です（maybe (1ul = 0ul) _ == emp）。

  • cas_box が失敗した場合は、lock_inv を復元して false を返すだけです。

with_invariants ブロックの外側では、CAS が成功していれば終了で、呼び出し側へ p を返します。そうでなければ再帰して再試行します。

演習

末尾再帰の acquire を while ループを用いて書き直しなさい。

ロックの解放

単純版であれば、ロックの解放はやや容易です。シグネチャは acquire の双対で、呼び出し側は p をロックに渡さねばなりません。

fn release (#p:slprop) (l:lock)
requires lock_alive l p ** p
ensures lock_alive l p
{
  unfold lock_alive;
  later_credit_buy 1;
  with_invariants l.i
    returns _:unit
    ensures later (lock_inv l.r p)
    opens [l.i] {
    later_elim _;
    unfold lock_inv;
    write_atomic_box l.r 0ul;
    drop_ (maybe _ _); //maybe release without acquire
    fold (maybe (0ul = 0ul) p);
    fold (lock_inv l.r p);
    later_intro (lock_inv l.r p);
  };
  fold lock_alive
}

この実装では、release は参照を無条件に 0ul に設定し、コンテキストに p があるので lock_inv を再証明します。

ただし、ロックがすでに解放状態であれば、すでに p を保持している可能性があります。解放済みロックをさらに解放すると、呼び出し側が資源をリークする恐れがあります。

演習

release を、解放前にロックが取得されるまでスピンするよう書き直しなさい。二重解放の回避策としては現実的ではありませんが、有益な練習になります。

演習

二重解放を防ぐためにロック API を再設計しなさい。1 つの方法は、取得時に解放許可を配布し、release がその許可を要求・消費する方式です。

演習

分数権限付きの存続（liveness）述語を追加し、ロックを割り当てて複数スレッド間で共有し、再び集約して、最終的に解放できるようにしなさい。
