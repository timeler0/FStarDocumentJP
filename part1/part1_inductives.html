帰納的データ型とパターンマッチ

この章では、F* で新しい型を定義する方法を学びます。これらは *帰納的型*（インダクティブ型）、あるいはより砕けて言えばデータ型と呼ばれます。さらに、これらの帰納的型に対する関数をパターンマッチで定義し、その性質を証明する方法も学びます。

注釈

ここでは帰納的型の最も基本的な形だけを扱います。特に、ここで示す型はインデックス付けや他の依存型の機構を用いません—それらは後の章に回します。

列挙型

すでに、unit はただ 1 つの要素 () をもつ型であり、bool は true と false の 2 要素をもつ型であることを見ました。

次のように、要素の列挙で独自の型を定義できます。

type three =
  | One_of_three : three
  | Two_of_three : three
  | Three_of_three : three

This introduces a new type three : Type, and three distinct constants One_of_three : three, Two_of_three : three, Three_of_three : three. These constants are also called "constructors" or "data constructors". The name of a constructor must begin with an uppercase letter.

注釈

この場合、各コンストラクタに毎回型 three を書くのは冗長に見えるかもしれません。実際、どれも three のコンストラクタです。このような場合、F* では次のように簡潔に書けます。

type three =
  | One_of_three
  | Two_of_three
  | Three_of_three

インデックス付きの型を使い始めると、各コンストラクタが定義した型の異なるインスタンスを生成しうるため、各コンストラクタの結果型を指定できることが重要になります。統一のため、本書では厳密には不要な場合でもコンストラクタに型注釈を付ける方針にします。

F* は、これらの値が互いに異なること、そして型 three の値はこれらだけであることを証明できます。

let distinct = assert (One_of_three <> Two_of_three /\
                       Two_of_three <> Three_of_three /\
                       Three_of_three <> One_of_three)

let exhaustive (x:three) = assert (x = One_of_three \/
                                   x = Two_of_three \/
                                   x = Three_of_three)

これらの新しい型を解析する関数は match 構文で書きます。F* の match は OCaml や F# とよく似ています。基本に馴染みがあるものとして進め、後ほどより高度な使い方も学びます。

いくつか例を示します。

let is_one (x:three)
  : bool
  = match x with
    | One_of_three -> true
    | _ -> false

let is_two (x:three)
  : bool
  = match x with
    | Two_of_three -> true
    | _ -> false

let is_three (x:three)
  : bool
  = match x with
    | Three_of_three -> true
    | _ -> false

判別関数（ディスクリミネータ）

これらの関数は、x : three が与えられたコンストラクタに一致するかを判定し、その都度 bool を返します。帰納的型の値がどのコンストラクタかを調べる関数は頻出なので、F* はそれらを自動生成します。例えば、次のように書く代わりに

let three_as_int (x:three)
  : int
  = if is_one x then 1
    else if is_two x then 2
    else 3

次のように書けます。

let three_as_int' (x:three)
  : int
  = if One_of_three? x then 1
    else if Two_of_three? x then 2
    else 3

言い換えると、帰納的型 t の各コンストラクタ T に対して、F* は T? という（判別関数）を自動生成し、v:t が T かどうかを判定できます。

網羅性（エグゾースティブネス）

もちろん、three_as_int をさらに直接的に書く方法もあります。

let three_as_int'' (x:three)
  : int
  = match x with
    | One_of_three -> 1
    | Two_of_three -> 2
    | Three_of_three -> 3

match を使うたび、F* はあらゆる場合を処理しているか（網羅しているか）を証明しようとします。上の three_as_int から 1 つケースを抜いてどうなるか試してみてください。

F* の網羅性検査は意味論的な検査であり、SMT ソルバを用いてすべてのケースが適切に扱われていることを証明できます。例えば、次のように書けます。

let only_two_as_int (x:three { not (Three_of_three? x) })
  : int
  = match x with
    | One_of_three -> 1
    | Two_of_three -> 2

引数の精錬により、その分岐には到達しえないため、パターン中の Three_of_three ケースは不要であることを F* が証明できます。

タプル

列挙の次の段階として、他の型の 2 要素・3 要素・4 要素…からなる複合型（タプル）を定義します。方法は次のとおりです。

type tup2 (a:Type)  (b:Type) =
  | Tup2 : fst:a -> snd:b -> tup2 a b

type tup3 a b c =
  | Tup3 : fst:a -> snd:b -> thd:c -> tup3 a b c

tup2 a b の型定義は、任意の型 a : Type と b : Type に対し Tup2 : a -> b -> tup2 a b であることを述べます。すなわち x:a と y:b から Tup2 x y : tup2 a b を作るコンストラクタです。

他の tup3 や tup4 も同様です—束縛変数の型注釈は推論可能です。

これらは 1 つのケースしか持たない帰納的型なので、Tup2?・Tup3?・Tup4? といった判別関数はあまり有用ではありません。しかし、タプルの各要素を取り出す（射影する）方法が必要で、match で行えます。

let tup2_fst #a #b (x:tup2 a b)
  : a
  = match x with
    | Tup2 fst _ -> fst

let tup2_snd #a #b (x:tup2 a b)
  : b
  = match x with
    | Tup2 _ snd -> snd

let tup3_fst #a #b #c (x:tup3 a b c)
  : a
  = match x with
    | Tup3 fst _ _ -> fst

let tup3_snd #a #b #c (x:tup3 a b c)
  : b
  = match x with
    | Tup3 _ snd _ -> snd

let tup3_third #a #b #c (x:tup3 a b c)
  : c
  = match x with
    | Tup3 _ _ thd -> thd

射影子（プロジェクタ）

射影は頻出なので、F* は自動生成します。具体的には、任意のデータコンストラクタ T : x1:t1 -> ... -> xn:tn -> t に対し、次の関数を生成します。

  • T?.xi : y:t{T? y} -> ti

つまり T?.xi は、y:t が T である場合に、T x1 ... xn の xi 成分を返す関数です。

tup2 と tup3 の場合、次の射影子が得られます。

  • Tup2?.fst、Tup2?.snd

  • Tup3?.fst、Tup3?.snd、Tup3?.thd

タプルの構文シュガー

タプルは非常によく使われるため、FStar.Pervasives.Native.fst には 14 要素までのタプル型が定義済みです。したがって通常は tup2 や tup3 などを自前で定義する必要はありません。

FStar.Pervasives.Native のタプル型には構文シュガーが用意されています。

  • tup2 a b の代わりに a & b、tup3 a b c の代わりに a & b & c と書けます（最大 14 要素まで）。

  • Tup2 x y の代わりに x, y、Tup3 x y z の代わりに x, y, z と書けます（最大 14 要素まで）。

  • タプルから第 i 成分を取り出すには、x._1、x._2、x._3 などと書けます（要素数が少なくとも i の場合）。

とはいえ、5 要素以上のタプルを使うなら、次に見る レコード を定義する方がよいでしょう。大きなタプルでは各成分の意味を覚えにくいからです。

レコード

レコードは、各フィールドに任意の名前を付け、構築や射影のための特別な構文を備えたタプルだと考えられます。例を示します。

type point3D = { x:int; y:int; z:int}

let origin = { y=0; x=0; z=0 }

let dot (p0 p1:point3D) = p0.x * p1.x + p0.y * p1.y + p0.z * p1.z

let translate_X (p:point3D) (shift:int) = { p with x = p.x + shift}

let is_origin (x:point3D) =
  match x with
  | {z=0;y=0;x=0} -> true
  | _ -> false

  • レコード型は波括弧 {} で定義します。type point3D を参照してください。

  • レコード値も波括弧で構築し、各フィールドに代入します。フィールドは順不同で構いません。origin を参照。

  • レコードのフィールドへは p.x のようなドット記法でアクセスします。ドット積を計算する dot の例を参照してください。

  • レコードは with 記法にも対応しており、指定したフィールド以外を元のレコードと同一にした新しいレコードを作れます。つまり translate_X p shift は { x = p.x + shift; y = p.y; z = p.z } を返します。

  • レコードは値のパターンマッチにも使えます。例えば is_origin では、（順不同で）フィールドをパターンに照合しています。

Option 型

F* の標準ライブラリでよく使うもう一つの型が option 型で、存在しない可能性のある値を表現するのに便利です。

2 つの整数 x と y に対して x / y を行う関数を考えます。y=0 のときは定義できませんが、以下のようにその場合を除外して部分的に定義できます。（もちろん、y = 0 を禁じるように関数の定義域を精錬する方法もありますが、ここでは option 型の例示が目的です。）

let try_divide (x y:int)
  : option int
  = if y = 0 then None else Some (x / y)

let divide (x:int) (y:int{y<>0}) = x / y

多くの関数型言語と同様、F* には null 値はありません。値が存在しない可能性がある場合は通常 option 型を用い、存在しない場合を None、存在する場合を Some v で表します。

共用体（ユニオン）、either 型

FStar.Pervasives には either 型も定義されています（下記）。

type either a b =
  | Inl : v: a -> either a b
  | Inr : v: b -> either a b

型 either a b は、v:a をもつ Inl v か、v:b をもつ Inr v のいずれかを表します。つまり a と b のタグ付き共用体です。タグ ``Inl``（左側）か ``Inr``（右側）を解析し、その内部の値で計算する関数は簡単に書けます。例：

let same_case #a #b #c #d (x:either a b) (y:either c d)
  : bool
  = match x, y with
    | Inl _, Inl _
    | Inr _, Inr _ -> true
    | _ -> false

let sum (x:either bool int) (y:either bool int{same_case x y})
  : z:either bool int{ Inl? z <==> Inl? x}
  = match x, y with
    | Inl xl, Inl yl -> Inl (xl || yl)
    | Inr xr, Inr yr -> Inr (xr + yr)

same_case x y は、2 つのユニオンが同時に左か右か、同じ側にあるかを判定します。

そこで sum x y では、x と y が同じ側にあるという精錬のもと、（両方左・両方右の）2 つだけを扱えばよく、F* は網羅性も証明できます。左側では中身が boolean なので || で結合し、右側では中身が整数なので + で結合し、適切なタグを付けて返します。結果の型 z:either bool int{ Inl? z <==> Inl? x } は、結果が x``（ひいては ``y）と同じ側であることを示しています。結果型は z:either bool int { same_case z x } と書くこともできました。

リスト

ここまでの型は“形式上”は帰納的でも、コンストラクタが自分自身の型を参照してはいませんでした。ここで最初の本当の帰納的型、リストを扱います。

Prims にある list の定義は次のとおりです。

type list a =
  | Nil  : list a
  | Cons : hd:a -> tl:list a -> list a

list 型はすべての F* プログラムで暗黙に利用でき、コンストラクタには特別（標準的）な構文があります。

  • [] は Nil

  • [v1; ...; vn] は Cons v1 ... (Cons vn Nil)

  • hd :: tl は Cons hd tl

必要であれば、常に Nil や Cons を明示的に書いて構いません（例：Cons hd : list a -> list a の部分適用をしたいときなど）。

リストの長さ

リストに対する簡単な関数を書いてみましょう。まずは長さの計算からです。

let rec length #a (l:list a)
  : nat
  = match l with
    | [] -> 0
    | _ :: tl -> 1 + length tl

length 関数は再帰的で、型 a に対して暗黙に多相です。任意の l : list a に対し、length l は nat を返します。定義ではリストにパターンマッチし、[] に至るまで末尾に対して再帰的に length を呼び出します。

練習問題

演習ファイルは こちら。

2 つのリストを連結する関数 append の定義を示します。引数の長さの和に等しい長さのリストを常に返すことを型で証明できるよう、適切な型を与えられますか？

let rec append l1 l2
  = match l1 with
    | [] -> l2
    | hd :: tl -> hd :: append tl l2

解答

val append (#a:Type) (l1 l2:list a)
  : l:list a { length l = length l1 + length l2 }
